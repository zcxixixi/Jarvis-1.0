# Jarvis 核心产品原则

> ⚠️ **最核心的原则 - 必须严格遵守**

---

## 🎯 核心宗旨

### 1. 用户体验至上
```
以用户为导向
让用户舒服
每个功能都必须让用户感到舒服
```

### 2. 信息必须真实 ⚠️
```
所有信息必须来自真实数据源
绝对不允许编造、捏造、虚构任何信息
包括但不限于：新闻、邮件、数据、事实
```

**零容忍**：任何虚假信息 = 立即下线

---

## 开发准则

### 1. 功能取舍原则

```
如果不能让用户舒服 → 不做
宁可不做，也不能做出让用户不舒服的功能
```

### 2. 发布标准

```
任何功能上线前必须：
1. 多次用户测试
2. 确认用户开心
3. 验证用户舒适度
4. 只有通过以上测试才能发布
```

### 3. 测试优先级

```
优先级排序：
1. 用户体验测试 (最高优先级)
2. 功能正确性测试
3. 性能测试
```

---

## 实施细则

### 每个功能开发流程

```
1. 设计 → 问：这会让用户舒服吗？
2. 实现 → 问：用户用起来方便吗？
3. 测试 → 问：用户会喜欢吗？
4. 发布 → 问：我们确定用户会开心吗？
```

### 判断标准

**好的功能**：
- ✅ 用户不需要思考就能使用
- ✅ 用户感觉被照顾到
- ✅ 用户愿意反复使用
- ✅ 用户会推荐给朋友

**不合格的功能**：
- ❌ 用户需要学习才能用
- ❌ 用户感觉麻烦
- ❌ 用户用一次就不想再用
- ❌ 用户抱怨或不满

---

## 案例

### 良好示例
- 测试音量调低（避免吓到用户）✅
- 语音反馈温和友好 ✅
- 错误处理优雅不崩溃 ✅

### 需要改进
- 音频丢包导致卡顿 ❌ → 正在优化
- 测试音太大声 ❌ → 已修复

---

## 对所有贡献者的要求

**Antigravity 和 OpenCode 都必须**：
1. 每个 PR 前问：这让用户舒服吗？
2. 测试时站在用户角度
3. 收到负面反馈立即优先处理
4. 功能完成 ≠ 用户满意

---

> **记住**：技术再先进，如果用户不舒服，就是失败的产品
